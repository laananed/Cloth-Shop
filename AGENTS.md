# AGENTS.md

本文件用于约束在 `Cloth-Shop` 仓库中工作的 Codex、ChatGPT、Claude Code 或其他代码 Agent。

目标不是让 Agent 尽可能多地修改代码，而是让每次修改都形成一个可理解、可测试、可回滚的最小业务闭环。

## 1. 项目定位

- 项目：服装商城 / 进销存管理系统课程设计。
- 数据库：MySQL 8.0.28，数据库名 `frieren_cloth_shop_db`。
- 后端：FastAPI + PyMySQL。
- 前端：原生 HTML、CSS、JavaScript ES Module。
- 测试：Node.js 内置测试框架，主要测试文件为 `tests/site.test.js`。
- 默认分支：`master`。
- 当前后端端口：`8050`。
- 当前前端端口：`5900`。

课程设计重点是数据库结构、业务一致性、事务、库存、订单状态和可演示性。不要为了追求框架复杂度而破坏现有稳定链路。

## 2. 主要目录与职责

以下只是当前结构地图。执行任务前仍必须读取实际文件，实际代码优先于本文件描述。

- `index.html`：前台页面结构、购买弹窗、商品图片预览和个人中心。
- `admin.html`：后台登录、商品、订单、图片、SKU 和统计页面结构。
- `src/main.js`：前后台主要交互、状态协调和 API 调用入口。
- `src/styles.css`：前后台主要样式。
- `src/sku-utils.js`：结构化 SKU 组合、校验和辅助逻辑。
- `src/ranking.js`：销量排名逻辑。
- `src/product-ordering.js`：商品可售状态与前台排序逻辑。
- `src/content.js`、`src/account-store.js` 等：静态数据、旧本地状态或兼容逻辑。不要把已经迁移到数据库的业务重新改回 `localStorage` 或 mock 数据。
- `backend/app/main.py`：FastAPI 路由、请求模型、主要业务逻辑和数据库调用。
- `backend/app/db.py`：数据库连接和环境变量加载。
- `backend/requirements.txt`：后端 Python 依赖。
- `sql语句/`：数据库结构、视图、存储过程、触发器、测试数据与账号初始化 SQL；必须遵循当前 README 和架构文档记录的执行顺序。
- `tests/site.test.js`：前端结构、工具函数和关键回归断言。
- `docs/current-project-architecture.md`：当前项目结构、业务链路、功能完成度和已知缺口的架构快照。
- `docs/`：课程设计、开发计划和说明文档。
- `start_dev.ps1`、`start_dev.bat`：本地前后端一键启动。
- `backend/uploads/`：运行时上传文件。除非任务明确要求，不要把本地上传产物加入版本控制。
- `backend/.venv/`：本地后端虚拟环境，不提交到仓库。

## 3. 每次任务开始前必须完成的分析

在修改代码之前，先检查当前工作区和相关代码，不要直接根据任务标题开始编码。

至少完成以下步骤：

1. 阅读当前任务、本文件、`docs/current-project-architecture.md`、`README.md` 和相关目录中的补充说明。
2. 检查 `git status`、当前分支、当前提交和未提交差异，禁止覆盖用户已有修改。
3. 使用文件名、函数名、API 路径、数据库表名和 `data-*` 属性搜索真实调用链。
4. 明确当前系统已经具备什么，而不是假设项目仍处于旧版本。
5. 输出或在内部建立本次影响地图：
   - 当前行为；
   - 本次只完成的业务闭环；
   - 用户操作后应看到什么；
   - 数据库应发生什么变化；
   - 前端、后端、SQL、文档和测试分别涉及哪些文件；
   - 哪些现有功能不能破坏；
   - 自动测试与人工测试方法。
6. 对跨层功能，按下面方向追踪完整链路：

```text
页面入口
→ DOM/data-* 事件
→ src/main.js 或工具模块
→ HTTP 请求
→ FastAPI 路由与请求模型
→ SQL / 视图 / 存储过程 / 事务
→ API 响应
→ 前端状态更新与重新渲染
```

如果链路没有追踪完整，不要开始大范围修改。

## 4. 修改范围原则

- 优先完成一个最小闭环，避免把无关重构、UI 优化和新功能混入同一任务。
- 只修改实现当前闭环所必需的文件。
- 不要因为文件较大而整体重写 `src/main.js` 或 `backend/app/main.py`。
- 优先复用现有函数、状态映射、请求封装和 CSS 组件。
- 新增重复逻辑前，先搜索是否已有同类工具函数。
- 修复根因，不要只隐藏报错、吞异常或增加无条件兜底。
- 不得删除或降级未在任务范围内的功能。
- 不得擅自更换技术栈、端口、数据库、认证方式、目录结构或依赖管理方式。
- 不得擅自增加生产依赖。确实需要时，先说明原因、替代方案和影响。
- 不得引入真实支付、优惠券、多仓库、复杂 RBAC 等超出当前课程设计范围的系统。
- 不要修改或提交 `.env`、真实密钥、真实密码和本机数据库凭据。
- 保留中文界面和中文错误信息，避免产生乱码或 mojibake。
- 任务提示词明确限制可修改文件时，必须遵守该范围；确需扩大范围时，先说明原因和证据。

## 5. 数据库与 SQL 规则

- 数据库结构变更必须通过新的编号增量 SQL 表达，除非任务明确要求修复尚未发布的迁移。
- 不要为了方便直接修改已经执行过的历史 SQL，并假设其他环境会自动同步。
- 新迁移应说明：前置条件、执行顺序、影响表、是否可重复执行、回滚或恢复方式。
- 先检查表、字段、索引、视图、触发器和存储过程的现状，不得根据名称猜测结构。
- SQL 必须使用参数化查询，不要拼接用户输入。
- 多表写操作必须考虑事务、异常回滚和并发一致性。
- 订单、库存、销量、支付和退款相关变更必须作为一致性整体审查。
- 商品、SKU、图片、地址等业务数据默认沿用逻辑删除，不要无理由改成物理删除。
- 修改表字段时，检查所有相关视图、查询别名、序列化函数和前端字段映射。
- 修改订单状态时，检查创建、支付、取消、发货、取消发货、退款申请、退款审核和销量回滚链路。
- 修改库存时，检查可用库存、锁定库存、购物车有效性、直接购买、购物车结算和退款恢复。
- 不得声称 SQL 已通过真实数据库测试，除非本次确实连接数据库并执行了验证。
- `sql语句/04_测试数据与验证.sql` 会修改测试数据，只能在明确的测试数据库中执行；不得未经确认在重要数据库中运行。

## 6. 后端规则

- FastAPI 请求参数应通过 Pydantic 模型或明确的 `Form`、`File`、`Header` 参数校验。
- 对外错误应返回明确状态码和用户可理解的中文 `detail`。
- 管理员接口必须继续校验管理员令牌和权限。
- 不要在新接口中绕过现有认证、库存校验或订单状态校验。
- API 字段变化应尽量向后兼容；必须破坏兼容性时，要同步修改所有调用方、架构文档和测试。
- 数据库异常必须回滚，并转换为可诊断的接口错误；不要静默失败。
- 不要让通用 `except Exception` 覆盖已有 `HTTPException` 的状态码和错误信息。
- 上传文件必须校验格式、大小和空文件，并在数据库写入失败时清理已保存文件。
- 修改 CORS、端口或静态文件挂载前，先检查前端启动方式、README、启动脚本和现有测试。
- 使用项目虚拟环境执行后端命令，优先使用 `backend/.venv/Scripts/python.exe`，避免误用全局 Python。

## 7. 前端规则

- 保持前台和后台现有视觉方向，除非任务明确要求 UI 重构。
- 使用现有 `data-*` 属性体系绑定交互，不要无理由改名并破坏测试或选择器。
- 数据库已经接管的购物车、订单、地址、后台商品和后台订单功能，不得重新以 mock 或 `localStorage` 作为权威数据源。
- 对加载中、空数据、接口失败、401/403、库存不足、商品下架和 SKU 无效状态提供明确反馈。
- 按钮禁用必须同时有业务校验，不能只依赖视觉上的 `disabled`。
- SKU 相关功能必须使用真实 `sku_id`，并保持颜色、尺码、价格、库存和在售状态一致。
- 修改商品图片时，检查主图、图片列表、购买弹窗、全屏预览和后台图片管理。
- 修改订单显示时，统一使用现有状态格式化逻辑，避免前后台出现不同文案。
- 不要强制移动端屏幕方向；页面应允许用户通过设备方向自然切换。
- 对新增纯逻辑优先抽取小型工具模块并增加单元测试，但不要为简单功能过度拆分。

## 8. 大型任务与多 Agent 协作

只有当任务可以清晰拆成互不依赖的分析部分时才使用子 Agent。

推荐的只读分析分工：

- 前端 Agent：页面入口、事件绑定、渲染状态和用户可见行为。
- 后端 Agent：API、请求模型、认证、事务和错误处理。
- 数据库 Agent：表、索引、视图、存储过程、触发器和数据迁移。
- 测试 Agent：现有测试覆盖、回归风险和可执行验证命令。
- 文档 Agent：README、架构文档和课程资料与真实代码的一致性。

主 Agent 必须汇总各方结果，形成统一影响地图后再修改。

禁止：

- 多个 Agent 同时修改 `src/main.js`、`backend/app/main.py` 或同一个 SQL 文件。
- 子 Agent 在未说明范围时自行重构。
- 只汇总子 Agent 结论而不核对真实代码。
- 让子 Agent 把历史计划或旧文档当作当前实现。

## 9. 测试要求

每个业务闭环完成后，执行与改动对应的最小测试，并尽量执行完整基础测试。

Windows 当前基础命令：

```powershell
npm.cmd test
node --check src\main.js
backend\.venv\Scripts\python.exe -m py_compile backend\app\main.py
backend\.venv\Scripts\python.exe -m py_compile backend\app\db.py
```

修改其他 JavaScript 模块时，补充：

```powershell
node --check <修改的 JavaScript 文件>
```

按改动类型增加验证：

- 前端结构或样式：打开对应页面，检查桌面端和移动端关键交互。
- API：通过 `/docs` 或等价请求验证成功、参数错误、权限错误和业务冲突。
- 数据库迁移：记录执行 SQL、执行顺序、关键查询和预期结果。
- 订单链路：至少验证创建、支付或取消，以及相关库存变化。
- 退款链路：验证申请、同意或拒绝、库存恢复、销量回滚和状态显示。
- SKU 链路：验证组合生成、唯一性、上下架、库存、购买、购物车和订单明细。
- 图片链路：验证上传、主图、追加、逻辑删除、前台预览和文件清理。
- 管理员链路：验证登录、刷新状态、401/403、退出和后台数据清理。
- 启动链路：验证 `start_dev.ps1` / `start_dev.bat`、后端 `8050`、前端 `5900`、CORS 和虚拟环境路径。

不得使用“应该通过”代替真实结果。无法执行的测试必须明确写为“未执行”，并说明原因和人工验证步骤。

源码字符串或正则断言只能证明结构契约存在，不得描述成完整 API、数据库或浏览器端到端测试。

## 10. 完成标准

任务只有同时满足以下条件才算完成：

- 用户可见行为符合需求。
- 数据库状态符合业务规则。
- 前端、API、SQL 和测试之间字段一致。
- 未破坏现有关键链路。
- 自动测试通过，或明确记录未能执行的测试。
- 没有遗留调试输出、临时代码、测试账号硬编码或无关文件。
- README 或相关文档在启动方式、迁移顺序、接口、权限或架构状态发生变化时得到同步更新。
- `docs/current-project-architecture.md` 已按第 14 节规则更新，或明确说明无需更新。

## 11. 最终回报格式

每次完成修改后，向用户返回：

1. 本次完成的闭环。
2. 修改文件列表及每个文件的作用。
3. 用户实际如何感受到变化。
4. 数据库或 API 发生的变化。
5. 执行过的测试命令及逐项结果。
6. 人工验收步骤。
7. 未验证事项、风险和后续建议。
8. 建议的 Git commit 名称。
9. `docs/current-project-architecture.md` 是否更新、更新了哪些章节，或无需更新的原因。
10. 当前 `git status --short` 和 `git diff --stat` 摘要。

不要只返回“已完成”或测试数量。

## 12. Git 与提交规则

- 不要覆盖、回滚或格式化用户未提交的无关修改。
- 禁止使用破坏性命令，例如无授权的 `git reset --hard`、强制推送或删除分支。
- 一个提交对应一个可测试的业务闭环。
- 普通任务默认不自动提交、不自动推送；只有当前任务明确授权时才执行。
- 多阶段任务明确要求自动提交时，每个阶段测试通过后创建独立最小提交。
- 只有用户明确要求推送时才 `push`。
- 提交前优先使用精确路径 `git add <file>`，不要默认使用 `git add .`。
- 建议提交信息使用简洁的 Conventional Commit 风格，例如：

```text
feat: 完善商品 SKU 管理闭环
fix: 修复退款后库存未恢复问题
test: 补充订单状态回归测试
docs: 更新数据库迁移与验收说明
```

## 13. 任务冲突时的优先级

发生冲突时按以下顺序处理：

1. 用户当前任务中的明确要求。
2. 当前目录更近的 `AGENTS.override.md` 或 `AGENTS.md`。
3. 本文件。
4. `docs/current-project-architecture.md`、README 和其他项目文档。
5. Agent 的通用偏好。

任何情况下都不得以“优化”为理由绕过数据安全、权限、事务、测试或用户明确限制。

## 14. 当前架构文档读取与维护规则

项目的当前架构快照位于：

`docs/current-project-architecture.md`

### 14.1 每次任务开始前

每次分析、修改或审查项目时，必须先读取：

1. 当前任务提示词；
2. 根目录 `AGENTS.md`；
3. `docs/current-project-architecture.md`；
4. `README.md`；
5. 本次任务涉及的真实代码、SQL 和测试。

`docs/current-project-architecture.md` 用于快速了解项目结构、功能边界、核心调用链和已知缺口，但它只是某个时间点的架构快照。

不得只根据架构文档修改代码。

如果架构文档、README、历史计划与当前代码冲突，以当前真实代码、SQL 和实际运行结果为准，并在任务完成后修正文档。

### 14.2 必须更新架构文档的情况

本次任务发生以下任一变化时，必须最小更新 `docs/current-project-architecture.md`：

- 新增、删除、移动或拆分核心文件；
- 新增、删除或修改 API 方法、路径、请求字段、响应字段或权限；
- 数据库表、字段、索引、外键、视图、存储过程、触发器或函数变化；
- SQL 初始化顺序或迁移方式变化；
- 前端或后端端口、启动脚本、虚拟环境位置、CORS 或静态文件挂载变化；
- 数据权威来源发生变化，例如从 `localStorage` / mock 迁移到数据库；
- 管理员认证、普通用户权限或资源归属策略变化；
- 新增、修复或删除一个核心业务闭环；
- 某项功能在“已完成、待验收、尚未完成”之间发生状态变化；
- 自动测试命令、测试数量或覆盖范围发生变化；
- 架构文档中记录的已知缺陷已经修复，或发现新的明确缺陷。

### 14.3 通常不需要更新的情况

以下修改在不影响架构、接口和功能状态时，通常不更新架构文档：

- 纯文案修正；
- 颜色、间距、字体等局部样式调整；
- 不改变行为的代码格式化；
- 局部变量改名；
- 小型空值保护；
- 只修复测试自身路径且不改变项目结构；
- 不改变调用链的内部实现优化。

如果不更新，最终回报必须明确说明：

`架构文档更新：无需更新`

并说明原因。

### 14.4 更新方式

更新架构文档时必须遵守：

- 只修改与本次任务相关的章节，不整体重写文档；
- 删除已经失效的缺陷描述；
- 更新受影响的文件、函数、API、数据库对象和业务流程；
- 更新相关功能的完成状态；
- 测试数量和结果只能填写本次真实执行结果；
- 未执行真实数据库、HTTP 或浏览器测试时，必须明确写“未执行”；
- 不得把计划、推测或历史对话写成当前事实；
- 不得写入 `.env`、密码、令牌或其他秘密；
- 不得伪造尚未产生的 Git commit SHA。

文档头部的 commit 应视为“本次任务开始时的审计基线”。

在尚未提交当前修改时，可以保留起始 commit，并注明文档包含当前工作区待提交变更，不得填写未来 commit。

架构文档应与实现该变化的代码放在同一个业务闭环提交中。

### 14.5 任务结束时必须报告

每次最终回报增加一项：

- `架构文档更新：已更新 / 无需更新`
- 更新了哪些章节；
- 或无需更新的具体原因。

只有当前提示词明确要求不修改文档时，才可以跳过本规则，并在最终回报中说明。


## 15. 用户只提供需求和目标时的处理规则

用户不需要具备项目经理、架构师或测试工程师能力，也不要求用户主动写出完整的修改范围、非目标和保护范围。

当用户只提供“需求”和“目标”时，Agent 必须先根据以下信息自行补全任务边界：

1. 当前任务提示词；
2. `AGENTS.md`；
3. `docs/current-project-architecture.md`；
4. `README.md`；
5. 本次相关真实代码、API、SQL 和测试；
6. 当前 Git 状态和未提交修改。

### 15.1 Agent 必须自行推导的内容

在开始修改前，Agent 应自行形成并明确说明：

- 当前系统已经具备的相关能力；
- 本次最小可交付业务闭环；
- 建议修改范围；
- 建议暂时不做的内容；
- 必须保护的现有功能；
- 可能涉及的前端、后端、数据库、文档和测试文件；
- 用户可见结果；
- 数据库或 API 的预期变化；
- 自动测试和人工验收方法；
- 是否需要更新 `docs/current-project-architecture.md`。

不得因为用户没有写出这些内容就停止工作，也不得要求用户先学会拆分技术任务。

### 15.2 默认推导原则

Agent 推导任务边界时遵守：

- 优先选择能够独立验收的最小闭环；
- 不顺便加入无关功能、重构或 UI 优化；
- 默认保护所有与本次需求无关的现有功能；
- 默认不修改数据库结构，除非需求无法在现有结构上正确实现；
- 默认不新增生产依赖，除非没有合理替代方案；
- 默认不改变端口、技术栈、认证方式和目录结构；
- 默认不清空、覆盖或批量修改现有业务数据；
- 默认不自动提交和推送；
- 默认只更新与本次任务直接相关的架构文档章节。

### 15.3 什么时候需要向用户提问

只有出现以下情况之一时，才需要在修改前向用户提问：

- 存在两种以上会明显改变用户体验的合理方案；
- 需求会删除或覆盖现有数据；
- 需求涉及不可逆数据库迁移；
- 需求会改变权限、支付、库存或订单状态规则；
- 用户目标与现有项目架构明显冲突；
- 无法判断某项已有功能是否必须保留；
- 缺少必要账号、接口、文件或环境，导致任务无法验证；
- 修改可能超出课程设计范围或引入明显过度设计。

问题应尽量少，并说明不同选择的影响。能够通过代码和文档确认的内容，不要反复询问用户。

### 15.4 开始修改前的边界确认

当用户只提供需求和目标时，Agent 在真正修改前应先输出一份简短的“任务边界确认”，至少包括：

```text
当前已有：
本次只完成：
用户完成操作后会看到：
数据库/API 将发生：
建议修改文件：
暂时不做：
必须保护：
测试方式：
架构文档：
```

如果用户没有反对，Agent 可以按该边界继续执行。

对于低风险、小范围、结论明确的 Bug 修复，可以在同一轮中完成边界确认后直接修改，不必等待额外确认。

### 15.5 最终回报

任务完成后，除第 11 节要求外，还应说明：

- 哪些任务边界是根据代码和文档自行推导的；
- 是否存在未向用户确认但采用的默认假设；
- 这些假设是否影响后续功能或数据。
